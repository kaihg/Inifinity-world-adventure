# E2E 觀察問題積壓（2026-06-30）

本文件記錄 2026-06-29 E2E 測試（封存世界→新世界→開場→2局）觀察到的問題，
屬於本次 ingest 重設計範圍**之外**的項目，留作後續追蹤。

---

## 問題 1：屬性系統數值範圍設計

**所在位置**：`world/setting.md` `## 屬性系統`

**現象**
- 數值範圍分段：1–5（弱）、6–20（普通）、21–40（強）、41–70（頂尖）、71+（神域）
- 越往後區間越大，導致「普通」只有 15 格但「頂尖」有 30 格，比例失衡
- 敘事中出現「林逸力量 9，位於普通下限」這類直接引用數字的描寫，沒有鋪墊就讓主角「知道」自己的屬性數值意義

**問題**
1. 數值範圍的比例設計是否合理？是否刻意留給後期成長空間？
2. 敘事直接引用屬性數字偏向「網遊介面 HUD 化」，與「黑暗寫實奇幻」的基調有落差
3. 主角沒有被明確告知屬性系統的意義，卻能在敘事中無縫理解，缺少過渡說明

**建議方向**
- 若想維持寫實風格，屬性數字應只出現在系統介面回覆（如晶體讀出的聲音），主角對「力量 9 是什麼意思」應有疑惑反應，而非直接理解
- 或調整設定：系統主動給出「你的各項數值與一名受過基礎訓練的士兵相當」之類的文字說明

## 問題 2：`secrets.md` 生成範圍太 meta

**所在位置**：`world/dungeons/<dungeon-id>/secrets.md`（進入副本時由 dungeon 模組生成）

**現象**
- 生成的 `bio-hazard-raccoon-city/secrets.md` 直接把「主神真實動機」、「主神與副本的共謀關係」、「如何反嗜主神」全部寫進第一個副本的隱藏真相
- 還設計了「對著母體低語『我選擇不成為神』即可摧毀主神」的反噬機制
- 套用了大量生化危機原作（吉爾、威斯克）但對其動機做了完全顛覆的改寫

**問題**
1. 第一個副本就能反噬主神，破壞了整個長期遊玩的懸念節奏
2. 「主神的真實動機」本應屬於 `world/gm-notes.md`（世界層隱藏設定），不應委託副本的 `secrets.md` 自行生成
3. 對原作設定的大幅顛覆（威斯克是「系統守護者」）風險較高，第一個副本的隱藏真相應以接近原作的秘密為主（例如「地下實驗室存在更早期的病毒變種」）
4. 整體違反了 CLAUDE.md 的原則：「隱藏設定逐步揭露」

**建議方向**
- `secrets.md` 的生成 prompt 加約束：
  - 副本隱藏真相必須以**原始 IP 設定**為主要素材
  - 禁止在副本 secrets 引入跨副本的元資訊（主神動機、執行者制度的真相等）
  - 跨副本的大後期揭露應由 `world/gm-notes.md` 負責，secrets.md 只管該副本自己的圈子

## 問題 6：`journal_summary.md` 缺少 opening 摘要

**所在位置**：`world/journal_summary.md`

**現象**
- `journal_summary.md` 目前只有 3 條記錄（Turn 1、Turn 2、Turn 3），缺少世界初始化時生成的 opening 敘事段落
- Opening 的 heading 是 `## [2026-06-29] 新世界啟用`，但 summary 第一條是 Turn 1 開始

**問題**
- 未來 resume 時 summary 會缺失「主角為何進入主神空間」的最初脈絡
- 若 journal.md 持續 append、summary 被用來壓縮 context，opening 的缺失會累積成資訊遺失

**建議方向**
- journal-summary 模組在生成摘要時，應掃描從「新世界啟用」段落起的所有內容，包含 opening

## 問題 7：journal.md 兩段落之間主角背景矛盾

**所在位置**：`world/journal.md`（第 1 段 vs 第 2 段）

**現象**
- 第 1 段（世界啟用 opening）：林逸是「物理系大學生」，在「深夜的校書館自習室」
- 第 2 段（Turn 1 玩家回合）：林逸是「剛考上醫學系卻被現實逼得去當夜班護工的學生」，在「醫院急診室」

兩段描述的出發場景、科系、身份完全不同，顯然是兩次模型呼叫用了不同設定生成的。

**根因**
- World init（`/api/world/init`）在生成 opening 敘事時，讀取的是 WorldSetupWizard 收集的設定（當時填的是物理系學生）
- Turn 1 由玩家直接發動，turn engine 讀取 `now.md` + `protagonist.md` 並重新產出背景，但模型在沒有強力錨點下自由發揮，生成了醫護工背景
- `protagonist.md` 裡的「背景」欄位文字與 opening 不一致，沒能鎖住後續回合

**建議方向**
- 主角背景的文字描述應在 opening 生成後**立刻同步回 `protagonist.md`**（或由 ingest 管線的主角 rewrite 保留敘事已建立的設定），確保後續回合的 narrative model 以相同的出發場景為錨
- 另外 `journal.md` 的 section header `[date]` 沒有實質意義（所有回合都是同一天），改成使用 `datetime`（如 `2026-06-29T14:17`）或「第 N 回合」更清楚

## 問題 8：`player-decisions.md` 設計問題

**所在位置**：`world/player-decisions.md`

**現象**
```
## 06f17957-a7ae-468e-9b31-803be10c8075
- 時間：2026-06-29T11:46:32.409Z
- 主角代數：1
- 玩家輸入：
```
- UUID 無法與 journal.md 對應 join（journal 沒有 UUID 索引）
- 「玩家輸入」欄位為空白（沒有寫入實際的玩家行動文字）
- 主角代數、時間戳對玩家可讀性低

**建議方向**
- 若此檔案是給引擎內部用的（例如記錄決策以便後處理），應直接放在 `.recall-index/` 或其他 derived 目錄，而非 `world/`
- 若要保留在 `world/`，建議改成人類可讀格式：`- [T1 2026-06-29T14:17] 觀察四周環境，嘗試向主神晶體詢問...`

## 問題 9：`setting.md` 世界 UUID 污染敘事

**所在位置**：`world/setting.md` L3、`world/journal.md` L69

**現象**
- `setting.md` 第 3 行：`- 世界 UUID：702157d6-507a-457c-964c-5d8ef724d46f`
- 該 UUID 出現在 narrative context（setting.md 被讀入 Layer 1 prompt），模型把它當作主角的「執行者編號」用：
  `【執行者編號：702157d6-507a-457c-964c-5d8ef724d46f-001】`

**問題**
- 世界 UUID 是引擎內部識別符，不是故事世界的元素
- LLM 讀到這個 UUID 後「借用」為主角編號，但後綴 `-001` 是 LLM 自行加的，引擎不能追蹤這個編號
- 如果多個主角代數存在，編號邏輯將完全混亂

**建議方向**
- 世界 UUID 移出 `setting.md`，改存在 `world/meta.json`（或現有的 `meta/player.md`）等引擎讀取、不注入 narrative prompt 的位置
- 若故事確實需要「執行者編號」作為敘事元素，應在主角初始化時由引擎生成並寫入 `protagonist.md`，由 narrative model 從那裡引用
