文件的主要痛點都源自「純對話 AI 沒有記憶體」這個根本限制。我們的引擎架構已經繞過：

| 文件痛點 | 我們的解法 | 備註 |
|---|---|---|
| AI 無法隨機 | `engine/roll.ts` 伺服器端 crypto d100 真隨機，骰值注入 prompt，模型只能依序取用 | 文件花了大量篇幅討論偽隨機骰池，包括 Reddit 的 Ryax75 方案；我們完全不需要 |

## 二、值得參考的觀點

### 2.1 文本渲染過濾器（去 AI 味）— 最大缺口

**文件原文：**「設計文本渲染過濾器：指定口吻，禁忌句型與文法，列出例句。」「規範文字風格最有效控制 AI 對話性格，因為 AI 非常非常容易受到玩家的說話習慣影響。」

**現狀問題：**

Layer 1 的 system prompt（`app/src/engine/turn/prompts.ts:87-123`）Iron Laws 全部都是**結構性約束**：
- 第三人稱
- 不能揭露秘密
- 不能問玩家下一步
- 不能自編骰值
- ……

但**完全沒有任何文風指令**——沒有指定口吻、沒有禁忌句型、沒有正面風格定調。模型在沒有文風約束的情況下，會回到 RLHF 訓練出來的預設「AI 腔」。

**行動建議：**

在 Layer 1 prompt 加入一個 `## 文風` 區塊，包含：

1. **正面風格定調** — 定義我們要的敘事調性（需配合 `setting.md` 的世界基調）
2. **禁忌句型清單** — 明確列出禁止使用的 AI 慣性句型/結構
3. **需要看實際產出才能精準設計** — 不同 model provider 的 AI 味表現不同，需要根據我們實際使用的模型產出樣本，歸納出該模型的具體慣性模式，而非套用泛用清單

> **待辦**：收集實際劇情產出中「覺得像 AI 在寫」的段落，歸納具體模式後再設計禁忌清單。

### 2.2 斷言式指令 — 砍掉解釋性語句

**文件原文：**「不要講道理」「不需要教它『因為玩家快死了，為了不讓遊戲直接結束，所以……』，要使用斷言式指令」

**現狀：**

我們的 Iron Laws 有些地方帶了解釋語氣，例如（`prompts.ts:102`）：

```
敘事結尾不可詢問玩家下一步要做什麼、不可徵求玩家指示或列出選項；
把敘事當成寫一部完整小說的段落收尾，建議行動由另一支獨立流程處理，與你無關。
```

後半句在解釋「為什麼」。模型不需要知道原因，更精練的版本：

```
敘事結尾禁止詢問玩家方向、禁止列舉選項。
```

類似的地方還有幾處（L2 的 `awaiting_user_input` 說明、L1 的骰值規則解釋等），可以在設計文風區塊時一併精練。

**效益**：省 token + 減少模型注意力分散。不過優先級低於 2.1，因為這些解釋性語句不會直接造成「AI 味」。

### 2.3 數值程度化

**文件原文：**「不要教 AI 用數值，直接教它數值程度化。讓它使用 [未涉獵/入門/初學/略通/熟練/專精/精通]，而不是技能等級 1-7，這樣它才懂得將變數融入故事裡。」

**關聯**：這不是 prompt 的問題，而是 `world/setting.md` 和 `protagonist.md` 的設計決策。如果角色屬性用純數值表示，模型在敘事中傾向生硬地報數字。改用語義等級後，模型自然能織入敘事（「他的劍法已達熟練境界」vs「他的劍法 Lv.4」）。

**待確認**：需要檢查目前 setting/protagonist 的屬性格式。

### 2.4 負向提示的注意事項

**文件原文：**「AI 通常會特別注意那些不能做的事，也因此當寫到禁止事項的時候，需要特別注意規則衝突。」

這點在設計禁忌句型清單時要注意：禁止某個句型時，要確保不會跟其他規則衝突（例如禁止「彷彿」但某些場景的比喻確實需要）。禁忌清單應該禁止的是**模式**（每段都用、公式化套用），而不是絕對禁止某個詞彙本身。

## 三、KV Cache 與 Prompt 排序最佳化分析

### 3.1 KV Cache 機制簡述

LLM 推論時，transformer 的 self-attention 層會為每個 token 計算 Key-Value 對。KV cache 的核心原理：**如果 prompt 的前綴（prefix）與上一次呼叫相同，已計算的 KV 對可以直接重用，不需重新計算**。

不同 provider 的快取機制：
- **Anthropic**：顯式 `cache_control` 斷點標記
- **OpenAI / 相容 API**：自動前綴快取（longest common prefix）
- **自架 vLLM / Ollama**：多數支援 prefix caching，行為類似 OpenAI

共同原則：**穩定不變的內容越集中在 prompt 開頭，可快取的前綴越長，後續呼叫的計算量與延遲越低**。

### 3.2 目前 Layer 1 的 prompt 排列順序

以 `buildMainSpaceMessages`（`prompts.ts:87-123`）為例，目前組裝順序：

```
 [1] 角色定義（穩定）        ← 每回合相同
 [2] ## 鐵則（穩定）          ← 每回合相同
 [3] ## 輸出格式（穩定）      ← 每回合相同
 [4] ## 本回合骰值（每回合變動）  ← ⚠️ 在這裡打斷了 cache prefix
 [5] ## 世界設定（穩定）      ← settingText 很少改，可能數千 token
 [6] ## 當前局勢（每回合變動）  ← now.md 快照
 [7] optional blocks（每回合變動）← intents, wiki, recall, nudge, pacing
 [8] ## 開場回合特別指引（僅首回合）
```

**問題**：骰值 `[4]` 每回合都不同，把快取前綴從 `[1-3]` 就截斷了。但 `[5]` 世界設定是穩定的大區塊，被夾在兩個變動區塊之間，白白浪費了快取潛力。

### 3.3 建議的最佳化排列

```
 [1] 角色定義（穩定）          ┐
 [2] ## 鐵則（穩定）            │
 [3] ## 輸出格式（穩定）        │ 可快取前綴
 [4] ## 世界設定（穩定）        │ ← 從 [5] 上移到骰值之前
 [5] ## 文風（穩定，新增）      ┘ ← 新區塊，放在快取前綴末端
 ─── cache break ───
 [6] ## 本回合骰值（每回合變動）  ← 從 [4] 下移
 [7] ## 當前局勢（每回合變動）
 [8] optional blocks（每回合變動）
 [9] ## 開場回合特別指引（僅首回合）
```

**效益**：
- 可快取前綴從 ~200-300 token（角色定義 + 鐵則 + 輸出格式）擴展到包含世界設定 + 文風，可能達 **1000-2000+ token**
- 世界設定在整個遊戲生命週期內極少變動，是最理想的快取內容
- 文風區塊一旦設計完成也是穩定的，適合放在快取前綴中

### 3.4 KV Cache vs 注意力曲線的取捨

HackMD 文件提到 AI 的注意力呈 U 型（開頭和結尾最強，中間最弱），建議「文本渲染過濾器放在最後」。

但這與 KV cache 最佳化有張力：
- **注意力曲線**：文風放末尾 → 注意力強
- **KV cache**：文風放穩定前綴（靠前）→ 可快取

**分析**：
1. 注意力曲線是在 prompt **非常長**（數千 token 的中段被遺忘）時才顯著。我們的 system prompt 總長度有限，「lost in the middle」效應不會太嚴重。
2. KV cache 的效益是**每回合都省**的計算量和延遲，對遊玩體驗有直接影響。
3. 我們的鐵則已經在開頭（注意力第一高峰），文風放在鐵則之後、變動內容之前，距離開頭不遠，仍在注意力的高區間。

**結論**：優先 KV cache，將文風區塊放在穩定前綴的末端（鐵則之後、骰值之前）。如果後續測試發現文風遵從度不足，再考慮在變動區塊末尾加一行簡短的文風提醒（重複核心禁忌）。

### 3.5 Layer 2 / Layer 3 的排列

同樣的原則適用：

**Layer 2（fast-control）**目前排列：
```
 [1] 角色定義（穩定）
 [2] 鐵則（穩定）
 [3] 輸出格式（穩定，且很長）
 [4] 骰值（變動）          ← 打斷 cache
 [5] 現有副本 id（變動）
 [6] 世界設定（穩定）      ← 被夾在變動區塊間
 [7] 副本 id（變動）
 [8] 當前局勢（變動）
 [9] 本回合敘事散文（變動）
```

建議同樣把 `[6] 世界設定` 移到 `[3]` 和 `[4]` 之間，擴大可快取前綴。

**Layer 3** 的各子 prompt 較短且每次都帶不同 entity 內容，快取空間有限，優先級最低。

### 3.6 副本 prompt 的額外考量

`buildDungeonMessages`（`prompts.ts:131-171`）有副本特有的區塊：
```
 ...（同主空間穩定區塊）
 [4] 骰值（變動）
 [5] 世界設定（穩定）
 [6] 副本 wiki（同一副本內穩定，跨副本變動）
 [7] 副本 secrets（同一副本內穩定，跨副本變動）
 [8] 當前局勢（變動）
 [9] optional blocks（變動）
```

副本內連續回合，wiki 和 secrets 通常不會每回合都變。建議排列：
```
 [1-3] 穩定區塊（角色 + 鐵則 + 輸出格式）
 [4]   世界設定（穩定）
 [5]   文風（穩定）
 ─── 以下若同一副本連續回合，也可能命中 cache ───
 [6]   副本 wiki（副本內相對穩定）
 [7]   副本 secrets（副本內相對穩定）
 ─── 以下必定每回合變動 ───
 [8]   骰值
 [9]   當前局勢
 [10]  optional blocks
```

## 四、行動項目總結

| 優先級 | 項目 | 說明 |
|---|---|---|
| **P0** | 收集實際 AI 味樣本 | 從 journal.md / log.md 收集「讀起來像 AI」的段落，歸納該 model 的具體慣性模式 |
| **P1** | 設計 `## 文風` 區塊 | 根據 P0 的樣本分析，設計正面風格定調 + 針對性禁忌句型清單 |
| **P1** | 重排 prompt 區塊順序 | 將世界設定移到骰值之前，擴大 KV cache 可快取前綴 |
| **P2** | 精練 Iron Laws 措辭 | 砍掉解釋性語句，改為純斷言式 |
| **P3** | 檢查數值程度化 | 確認 setting.md / protagonist.md 的屬性格式是否適合敘事融入 |
