# 劇情節奏停滯偵測與 Nudge 注入設計

## 背景

LLM 驅動敘事在「無限恐怖」這種長線、自由行動的遊玩模式下，容易出現節奏鬼打牆：連續多回合沒有實質劇情/狀態進展（反覆描寫「觀察」、戰鬥卡住沒有傷害變化），或更隱蔽的一種——**玩家已經明確輸入升級/決斷意圖（例如「推門進去，做好戰鬥準備」），主敘事 LLM 卻仍迴避衝突，繼續鋪墊氛圍**，沒有兌現玩家的行動意圖。

目前的回合管線（`buildMainSpaceMessages`／`runTurnLoop`，見 `app/src/engine/turn/`）沒有任何停滯偵測或節奏控制機制，也沒有「跨回合暫存建議」的概念——`AUTO_ADVANCE_MAX`（`app/src/engine/turn/index.ts:55`）只是防止自動推進迴圈跑到無限長的安全閥，跟劇情有沒有進展無關。

## 已否決的方向（及原因，避免重新討論同一條死路）

- **每回合都跑一個獨立判斷 LLM**：同步延遲在自動推進迴圈裡是乘法累積（迴圈跑 4 回合就多 4 次呼叫），且批判審查發現它跟現有「nudge 要插在哪一回合」的時序對不上——若放在迴圈外才注入，玩家輸入可能已經改變方向，nudge 反而誤導。
- **用 `now.md` 的 `chapter` 欄位當「是否合理停滯（例如 boss 戰）」的豁免依據**：不可靠。Schema 裡沒有 boss/小怪分類欄位，且角色戰力會隨遊玩膨脹，同一個副本表面 boss 前期是硬仗、後期可能變雜魚，標籤本身會失真。
- **新增一個欄位給敘事 LLM 自己回報「有沒有進展」**：自報式判斷有動機問題（LLM 不會誠實承認自己在水戰鬥），且重複輸出新欄位是多餘的——`commit_summary`（`app/src/engine/schema.ts:47-56`）本來就是每回合必填的一句話敘事摘要，已經逐回合進 git log，可以直接重用，不需要新欄位、不需要額外 append 動作。

## 目標

在 `runTurnLoop` 每輪迭代裡，於組裝**當前這一回合**的主敘事 prompt 之前，呼叫一個獨立的、有狀態的「nudge service」：

- 多數時候回傳空字串（無建議），幾乎零成本。
- 偵測到停滯時，回傳一句簡短的節奏建議文字，當場注入**同一回合**正在組裝的 `buildMainSpaceMessages()` system prompt 尾端。

關鍵設計原則：**nudge 的產生與消費在同一輪迴圈迭代內完成**，不存在「算出建議→存起來→等下一回合才能用」的跨回合搬運問題，因此不需要任何跨 HTTP request 的持久化狀態。

## Nudge Service 設計

### 狀態

- in-memory 滾動窗口，保存最近 N 筆 `commit_summary`（N 預設建議 5，可調），以及對應那一筆是「哪個回合、玩家輸入是什麼」的配對。
- **不寫入 `world/`，不進 git**：這是 heuristic 衍生狀態，不是 canonical truth。Server 重啟後窗口清空，需要重新累積 N 筆才能再次生效——這是可接受的退化（最差情況是重啟後前幾回合 judge 暫時失能），不是正確性問題。
- 不讀 `setting.md`/`gm-notes.md`/角色檔：避免無關背景稀釋判斷焦點，且這些原本就不是判斷「有沒有進展」所需要的資訊。

### 偵測邏輯（兩層，依信心程度分級）

1. **高信心規則（優先檢查，命中即觸發，不需要 LLM 二次確認）**：
   比對「上一回合玩家輸入」與「對應那一回合的 `commit_summary`」——若玩家輸入含明確升級/決斷意圖的動作詞（例如：推門、攻擊、戰鬥準備、追上去等），但對應回合的 `commit_summary` 沒有反映衝突/事件實際發生，視為高信心停滯訊號。
   理由：玩家已經主動表態要推進，敘事卻自己選擇迴避，幾乎不會是「合理的張力堆疊被誤殺」（張力堆疊是敘事主動鋪陳，不會在玩家明確要求衝突時還迴避）。

2. **低信心規則（輔助訊號，v1 可選，不強制做到自動觸發）**：
   比較最近 K 筆 `commit_summary` 的文字差異量（trigram overlap 或既有 recall 模組的本地嵌入 cosine 相似度，重用 `app/src/recall/` 基礎設施，不需要額外引入新的相似度計算依賴）。連續高度重複、沒有具體數值/事件變化（如「敵人HP 80%→60%」這類有差異的摘要不算重複）時，視為低信心停滯訊號。
   v1 範圍：低信心訊號可以先不做（只做規則1），待規則1上線觀察效果後再評估是否需要這條輔助規則；若要做，**禁止**在這條路徑上呼叫額外的判斷 LLM（成本/時序問題已在審查中確認，見上方「已否決的方向」），只做純規則式相似度比較。

### 輸出

- 空字串（預設、多數情況）。
- 一句簡短建議文字（命中規則時），措辭範例：「最近的行動意圖尚未被回應，這回合請讓衝突/事件實際發生」。建議文字模板化即可，不需要 LLM 生成。

### 觸發點

- 在 `runTurnLoop`（`app/src/engine/turn/index.ts`）每輪迭代裡，呼叫 `buildMainSpaceMessages()`（`app/src/engine/turn/prompts.ts:74`）之前，先問 nudge service 要不要附加建議。
- 每輪迭代結束、`commit_summary` 落地後，把這一筆連同對應玩家輸入餵回 nudge service 更新滾動窗口。

## 不變的部分

- `TurnControlSchema`（`schema.ts`）欄位定義不變，不新增欄位。
- 機率擲骰、raw 層落地、`now.md` 覆寫、wiki 提煉、commit 時機等下游邏輯完全不變。
- 不引入第三個獨立 LLM agent；nudge service 本身是純規則式（v1），不對外發起 LLM 呼叫。
- `character-pre-pass.ts`（NPC 意圖 pre-pass）不受影響。

## 錯誤處理與降級

- nudge service 任何內部錯誤（例如比對邏輯拋例外）都應該降級為「回傳空字串」，絕不能讓這個輔助機制影響主回合管線的正常產出。
- service 狀態（滾動窗口）僅存活於 process 記憶體；多 worker/多 process 部署時不共享狀態，每個 process 各自累積——這在目前單 process 架構下不是問題，若未來要水平擴展需重新評估（暫不在本次範圍內處理）。

## 測試策略

- 新增 nudge service 的單元測試：
  - 滾動窗口正確維護最近 N 筆。
  - 規則1（玩家意圖 vs 結果）：給定明確升級意圖輸入 + 無衝突的 `commit_summary`，應觸發；給定一般輸入或有衝突發生的摘要，不應觸發。
  - 空窗口（剛啟動/重啟後）行為：不應誤觸發。
- `turn.test.ts`：驗證 nudge service 回傳的建議文字確實出現在對應回合的 system prompt 裡；驗證 service 異常時不影響該回合敘事正常產出。

## 範圍外

- 不做低信心規則的 LLM 二次確認路徑（v1 不做，未來若規則1效果不足再評估）。
- 不做跨 process/跨重啟的持久化（窗口狀態允許隨重啟清空）。
- 不改變現有自動推進迴圈的終止條件（`awaiting_user_input`/`AUTO_ADVANCE_MAX`）。
- 不在 `now.md`/`threads` 等 canonical 文件裡記錄 nudge 觸發歷史。
